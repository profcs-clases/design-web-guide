01

Accesibilidad Web (WCAG)

La web es para todos

La accesibilidad no es opcional. Es un derecho y, en muchos países, una obligación legal. Además, beneficia a todos los usuarios.

Los 4 principios POUR

P

Perceptible

La información debe presentarse de formas que el usuario pueda percibir

O

Operable

Los componentes deben ser operables por todos los usuarios

U

Comprensible

La información y el manejo de la UI deben ser comprensibles

R

Robusto

El contenido debe ser interpretable por diversas tecnologías

Checklist esencial

Etiquetas en formularios

Cada input debe tener un <label> asociado

Textos alternativos

Toda imagen informativa necesita alt

Contraste adecuado

Ratio mínimo 4.5:1 para texto normal

Navegación por teclado

Todo debe ser usable sin mouse (Tab, Enter, Esc)

Estructura de encabezados

Un solo <h1>, jerarquía lógica h2→h3→h4

Atributos ARIA cuando sea necesario

aria-label, aria-describedby, role

Ejemplo: Botón accesible

Mal: sin contexto

<button>
    <i class="fas fa-trash"></i>
</button>

Bien: con label accesible

<button aria-label="Eliminar registro">
    <i class="fas fa-trash" aria-hidden="true"></i>
</button>

Herramientas de auditoría

Lighthouse

Integrado en Chrome DevTools. Auditoría completa de accesibilidad.

WAVE

Extensión de navegador que marca errores de accesibilidad.

axe DevTools

Análisis automatizado avanzado con sugerencias de corrección.

NVDA / VoiceOver

Lectores de pantalla reales. Pruébalos siempre.

Reto rápido

Desconecta tu mouse. Intenta navegar un sitio web que hayas hecho usando solo Tab, Enter y Esc. ¿Puedes completar las acciones principales? Si no, identifica 3 problemas.

02

Diseño Mobile-First Profundo

Pensar en pequeño primero

Mobile-first no es solo "hacer que se vea bien en el celular". Es una filosofía de diseño que prioriza lo esencial.

Principios mobile-first

Jerarquía estricta

Si no cabe en móvil, probablemente no es esencial

Touch-friendly

Objetos interactivos mínimo 44x44px

Velocidad primero

En móvil, cada KB cuenta

Gestos naturales

Swipe, pinch, tap (no hover)

Tablas en móvil: de horizontal a tarjetas

CSS 
/* En móvil: convierte tabla en tarjetas */
@media (max-width: 768px) {
    .table-cards table,
    .table-cards thead,
    .table-cards tbody,
    .table-cards th,
    .table-cards td,
    .table-cards tr {
        display: block;
    }
    
    .table-cards thead tr {
        position: absolute;
        top: -9999px;
        left: -9999px;
    }
    
    .table-cards tr {
        border: 1px solid var(--color-border);
        margin-bottom: 1rem;
        border-radius: var(--radius-md);
    }
    
    .table-cards td {
        border: none;
        position: relative;
        padding-left: 50%;
    }
    
    .table-cards td:before {
        content: attr(data-label);
        position: absolute;
        left: 1rem;
        width: 45%;
        font-weight: 600;
    }
}
HTML 
<!-- Uso: agrega data-label a cada celda -->
<table class="table-cards">
    <tr>
        <td data-label="Nombre">Juan Pérez</td>
        <td data-label="Cédula">V-12345678</td>
        <td data-label="Teléfono">0414-1234567</td>
    </tr>
</table>
Prueba siempre en dispositivo real. Las devtools del navegador son útiles, pero no reemplazan probar en un móvil real con conexión lenta.

Reto rápido

Toma una tabla de 5 columnas y conviértela en tarjetas para móvil usando la técnica de data-label.

03

Modo Oscuro / Claro

Respeta las preferencias del usuario

El modo oscuro no es solo una moda: reduce fatiga visual, ahorra batería en pantallas OLED y es una preferencia personal legítima.

Implementación con variables CSS

CSS 
:root {
    /* Tema claro (por defecto) */
    --bg-primary: #ffffff;
    --bg-secondary: #f9fafb;
    --text-primary: #111827;
    --text-secondary: #6b7280;
    --border: #e5e7eb;
}

/* Modo oscuro automático (preferencia del sistema) */
@media (prefers-color-scheme: dark) {
    :root {
        --bg-primary: #111827;
        --bg-secondary: #1f2937;
        --text-primary: #f9fafb;
        --text-secondary: #9ca3af;
        --border: #374151;
    }
}

/* Modo oscuro manual (clase en body) */
[data-theme="dark"] {
    --bg-primary: #111827;
    --bg-secondary: #1f2937;
    --text-primary: #f9fafb;
    --text-secondary: #9ca3af;
    --border: #374151;
}

body {
    background: var(--bg-primary);
    color: var(--text-primary);
}

Toggle de tema con JavaScript

JavaScript 
class ThemeManager {
    constructor() {
        this.theme = this.getTheme();
        this.applyTheme();
    }
    
    getTheme() {
        // Prioridad: localStorage > preferencia del sistema > claro
        const saved = localStorage.getItem('theme');
        if (saved) return saved;
        
        if (window.matchMedia('(prefers-color-scheme: dark)').matches) {
            return 'dark';
        }
        return 'light';
    }
    
    applyTheme() {
        document.documentElement.setAttribute('data-theme', this.theme);
    }
    
    toggle() {
        this.theme = this.theme === 'dark' ? 'light' : 'dark';
        localStorage.setItem('theme', this.theme);
        this.applyTheme();
    }
}

const themeManager = new ThemeManager();
document.getElementById('theme-toggle').addEventListener('click', () => {
    themeManager.toggle();
});

Consideraciones importantes

  • No uses negro puro (#000): usa grises oscuros (#111, #1f2937)
  • Reduce contraste: no uses texto blanco brillante
  • Imágenes: algunas pueden verse muy brillantes en modo oscuro
  • Sombras: en oscuro, usa bordes sutiles en lugar de sombras
  • Iconos: asegura que funcionen en ambos temas

Reto rápido

Implementa el toggle de modo oscuro con persistencia en localStorage. Asegúrate de que todo tu CSS use variables para que el cambio sea instantáneo.

04

Optimización de Rendimiento

Una interfaz lenta es una interfaz rota

Cada segundo de carga adicional reduce conversiones en 7%. En conexiones lentas, la optimización es crítica.

Imágenes: el enemigo #1

HTML 
<!-- Formato moderno -->
<picture>
    <source srcset="imagen.avif" type="image/avif">
    <source srcset="imagen.webp" type="image/webp">
    <img src="imagen.jpg" alt="Descripción" loading="lazy">
</picture>

<!-- Lazy loading nativo -->
<img src="foto.jpg" loading="lazy" alt="...">

<!-- Responsive images -->
<img 
    srcset="foto-400.jpg 400w, foto-800.jpg 800w, foto-1200.jpg 1200w"
    sizes="(max-width: 600px) 400px, (max-width: 1200px) 800px, 1200px"
    src="foto-800.jpg"
    alt="..."
>

CSS y JS optimizados

HTML 
<!-- CSS crítico en head -->
<style>
    /* Solo lo esencial para el primer render */
    body { margin: 0; font-family: sans-serif; }
    .header { ... }
</style>

<!-- CSS no crítico con preload -->
<link rel="preload" href="styles.css" as="style" 
      onload="this.onload=null;this.rel='stylesheet'">

<!-- JS al final o con defer -->
<script src="app.js" defer></script>

Métricas clave (Core Web Vitals)

Métrica Qué mide Objetivo
LCP Largest Contentful Paint < 2.5s
FID First Input Delay < 100ms
CLS Cumulative Layout Shift < 0.1

Herramientas de medición

  • Lighthouse - Auditoría completa (Chrome DevTools)
  • PageSpeed Insights - Análisis web de Google
  • WebPageTest - Pruebas desde múltiples ubicaciones
  • GTmetrix - Análisis detallado de rendimiento

Reto rápido

Ejecuta Lighthouse en una página web que hayas hecho. Identifica los 3 problemas principales de rendimiento y propón soluciones.
